1. Deepen the LLM service — stop the two adapters routing around the interface

Strong ports & adapters

app/services/llm/base.py · claude.py · ollama.py · factory.py · prompts.py · decks/worker.py · chat/service.py

Before — shallow interface

base.py — interface
claude.py
prompt fmt · JSON parse · card list fmt — duplicated
ollama.py
prompt fmt · JSON parse · card list fmt — duplicated

interface is a bare ABC stub; all behavior lives twice, below it

After — deep interface

base.py — owns prompt formatting, JSON parsing, error normalization, retries
claude.py
raw call only
ollama.py
raw call only

implementation moves up into the interface; adapters shrink to "make the request"

Problem — base.py is a bare ABC with three abstract stubs; card-list formatting, prompt-template filling, and JSON parsing are copy-pasted identically into claude.py and ollama.py, and each adapter leaks its own raw exception type past the seam. Two adapters justify a seam here, but nothing behind it is shared.

Solution — Move prompt formatting, JSON parsing, error normalization (one LLMServiceError), and retry policy into base.py as concrete methods the adapters call into; adapters implement only the transport call. Fold chat/service.py's manual system-prompt building into a `chat_with_context()` interface method so prompt engineering stops leaking past the seam.

  • Leverage: one interface, N adapters — a parsing fix lands for Claude and Ollama at once
  • Locality: prompt engineering concentrates in one seam instead of chat/service.py + two adapters
  • Tests: one contract suite runs against both adapters, no per-provider mock duplication

2. Give deck generation a real interface — collapse the pipeline into one deep module

After — one pipeline module

flowchart TD
  R[routes.py] -->|apply_async| T[generate_deck_task]

  classDef deep fill:#0f172a,color:#e2e8f0,stroke:#0f172a;

Problem — _run_generate_deck is a flat script coupling five steps directly to two service modules, with no interface of its own; failure-marking logic is copy-pasted between routes.py's pre-queue path and worker.py's mid-pipeline path, and SSE status strings ("searching_cards", "composing_deck") aren't enum-backed like the rest of TaskStatus.

Solution — Introduce a DeckGenerationPipeline.run(deck_id, task_id, prompt) module that owns the full sequence including failure handling, called identically from the Celery task and from routes.py's pre-queue guard. Promote the ad hoc progress strings into a shared enum published and consumed from one place.

  • Locality: one place owns "what happens when deck generation fails"
  • Leverage: tests call pipeline.run(), not a private free function
  • Tests: real Celery entrypoint becomes testable via eager mode against the interface

Problem — redis_cache.py's publish() is dead code in production — worker.py hand-rolls its own aioredis.from_url client and a bespoke _publish helper with a different message envelope and error-swallowing. The 24h cache TTL is also duplicated as separate constants in redis_cache.py and scryfall_service.py.

Solution — Extend redis_cache.publish to accept a dict payload (serialize once, in one place) and have worker.py call it instead of building its own client. Source the TTL constant from one module.

  • Locality: the pub/sub message envelope changes in one place, not two
  • Deletion test flips: deleting redis_cache.publish would now break worker.py, proving the module earns its keep

4. Move validation and error-mapping out of chat/routes.py, into the service it's misplaced above

Worth exploring in-process

app/chat/routes.py · service.py · core/guards.py

Before — layers inverted

routes.py — injection screening loop + Ollama error translation + HTTP mapping
service.py — prompt string building only

After — thin route, deep service

routes.py — maps typed errors → HTTP status
service.py — screening, provider calls, typed errors (ChatValidationError, ChatProviderUnavailable)

Problem — chat/routes.py holds the prompt-injection screening loop and Ollama-specific ConnectError/HTTPStatusError translation — the most interesting business logic in the module — while chat/service.py, the module named for it, only builds a string and unwraps JSON. The boundary is drawn backwards, and the untested Ollama error branches sit in the harder-to-unit-test route layer.

Solution — Move injection screening and provider-error translation into chat/service.py, raising typed exceptions; routes.py becomes a thin map from exception type to HTTP status, regardless of provider.

  • Leverage: a new provider or validation rule touches service.py only
  • Tests: provider-error branches become directly testable, not routed through FastAPI

Problem — the task:{task_id} pubsub channel name, and the raw SSE data: {...}\n\n framing, are each written independently in worker.py and tasks/routes.py with no shared constant and no test pinning the contract between publisher and subscriber.

Solution — a small shared task_channel(task_id) and format_sse_event(payload) helper, imported by both sides.

  • Locality: a channel-naming change can't drift silently between publisher and subscriber

Top recommendation

1. Deepen the LLM service

It's the highest-leverage seam in the app — every deck generation and every chat message crosses it — and it's currently the shallowest: two adapters duplicating prompt formatting, JSON parsing, and error handling with no shared contract test. Fixing it pays back at every call site and removes the asymmetric test coverage (Claude's invalid-JSON path is tested, Ollama's isn't) in one move.
